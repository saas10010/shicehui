# /tasks（练习任务）功能含义与解决问题

## 定位
- Web 教师端路由：`/tasks`（App Router：`app/(dashboard)/tasks/page.tsx` → `components/tasks/tasks-center.tsx`）。
- 原型实现：仅演示“创建任务/任务列表/状态徽标/完成统计占位”，数据持久化用 localStorage `shicehui:tasks`。

## PRD 对应
- `docs/Shicehui_PRD.md`：
  - FR13：为错题推荐同考点巩固题并生成练习任务下发。
  - FR14：学生端领取→作答→提交，记录完成状态与正确率；教师端可查看完成情况。
- `docs/Shicehui_UIUX.md`：流程 G（FR13/FR14）描述“推题下发与学生练习闭环”。

## 解决的实际问题（业务痛点）
- 批改后“错题→巩固练习”的转化成本高：教师需要手动找题、抄题/截图、逐个下发，效率低且难标准化。
- 巩固练习缺乏可追踪性：传统作业/口头布置难以量化领取率、完成率、正确率，教师无法快速评估补救是否有效。
- 学生练习缺少即时闭环：做完是否提交、是否掌握无法沉淀为数据，难以驱动后续教学决策。
- 家长监督入口分散：任务在家校沟通工具里容易被淹没，缺少清晰“下一步做什么”的可执行入口。

## 与小程序端的关系（原型）
- 家长/学生端小程序原型另有练习入口：`/mini/ps/tasks`（localStorage `shicehui:miniPsTasks`），体现“领取→作答→提交”的演示闭环。
